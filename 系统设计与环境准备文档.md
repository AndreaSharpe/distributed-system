# 商品库存与秒杀系统 - 设计与实现文档

为了完成分布式/微服务相关的课程要求，本系统（秒杀与库存管理）采用了微服务架构进行拆分。目前是项目的第一阶段，主要完成了基础服务的划分、数据库设计以及跑通最核心的登录下单链路。

## 一、 系统架构与服务拆分

目前系统暂时拆分为以下四个核心微服务，后续如果加网关（Gateway）再另行补充：

- **user-service（用户服务）**
  - 主要管用户注册、登录和鉴权。
  - 目前对外只暴露最基础的注册、登录（下发 Token）和获取个人信息的接口。
- **product-service（商品服务）**
  - 维护商品的元数据，比如名称、价格、描述和上架状态。
  - 负责给前端提供商品列表和详情展示。
- **stock-service（库存服务）**
  - 核心服务之一，单独拆出来是为了应对后面的秒杀高并发。
  - 提供查库存、预占库存、扣减以及释放库存的接口。
- **order-service（订单服务）**
  - 接收前端下单请求。
  - 内部需要串联校验用户信息、调用 stock-service 扣减库存，最后落库生成订单记录。

**大概的调用链路：**
客户端先通过 `user-service` 拿到 Token -> 带着 Token 访问 `product-service` 看商品 -> 发起抢购 -> `order-service` 收到请求后，内部通过 RPC/HTTP 调用 `stock-service` 预占库存 -> 预占成功则落库写订单 -> 返回下单结果。

> 注：具体的服务交互架构图我已经用 draw.io 画好了，源文件和导出的图片放在了项目根目录的 `docs/` 文件夹下。

## 二、 RESTful API 接口定义（v1.0初版）

第一版先把基础链路走通，所以接口字段比较精简，后续做秒杀防刷、限流的时候再加相关参数。

### 2.1 用户服务 (user-service)

**1. 用户注册**
- `POST /api/users/register`
- **Request:** `{"username": "alice", "password": "123456"}`
- **Response:**
  ```json
  {
    "code": 0,
    "message": "success",
    "data": { "userId": 1, "username": "alice" }
  }
  ```

**2. 用户登录**
- `POST /api/users/login`
- **Request:** 同上
- **Response:**
  ```json
  {
    "code": 0,
    "message": "success",
    "data": {
      "token": "xxxx-xxxx-uuid", 
      "userId": 1,
      "username": "alice"
    }
  }
  ```

**3. 获取当前登录用户信息**
- `GET /api/users/me`
- **Header:** `Authorization: Bearer <token>`
- **Response:** 返回当前登录用户的基础信息（脱敏）。

### 2.2 商品服务 (product-service)

**1. 查询商品列表**
- `GET /api/products?page=1&pageSize=10`
- **Response:** 返回带分页的商品数组（包含商品 ID、名称、价格、秒杀状态等）。

**2. 查询商品详情**
- `GET /api/products/{id}`

### 2.3 库存服务 (stock-service)

**1. 预占库存（防超卖核心）**
- `POST /api/stocks/reserve`
- **Request:**
  ```json
  {
    "productId": 1,
    "quantity": 1,
    "requestId": "uuid-for-idempotent" 
  }
  ```
- **说明:** 收到请求后先扣减可用库存，不马上扣总库存。

**2. 真实扣减与释放库存**
- `POST /api/stocks/deduct`：订单支付成功后调用，真正扣减库存。
- `POST /api/stocks/release`：预占后超时未支付，或者订单取消时调用，把库存加回去。

### 2.4 订单服务 (order-service)

**1. 创建订单**
- `POST /api/orders`
- **Header:** 必须携带 Token。
- **Request:**
  ```json
  {
    "productId": 1,
    "quantity": 1,
    "seckill": true 
  }
  ```
- **Response:** 返回生成的 `orderId` 和订单状态。

## 三、 数据库设计

数据库目前用的是 MySQL，统一建在 `seckill_db` 库里（前期为了方便本地调试，几张表先放在了一个库里，后续按微服务拆库随时可以迁移）。

包含 4 张核心表：
1. **user（用户表）**：主要存 `username` 和 `password_hash`（密码在代码里用了加盐哈希，不存明文）。
2. **product（商品表）**：存商品名、价格，后续为了秒杀会加上 `start_time` 和 `end_time`。
3. **stock（库存表）**：
  - `total`：总库存。
  - `available`：可用库存（预占时扣这个）。
  - `version`：加了这个字段用来做**乐观锁**，并发更新时防超卖。
4. **order（订单表）**：存 `user_id`、`product_id`、数量、金额和 `status`（已创建/已支付/已取消）。

> 各表之间的实体关系和外键 ER 图，同样保存在了 `docs/` 目录下。

## 四、 技术栈选型

考虑到开发效率以及自己对 Java 生态比较熟悉，项目用了下面这套技术栈：

- **核心框架**：`Spring Boot`（版本跟着课程要求走，目前用的 JDK 17）。起步依赖搞起来快，省去一堆配置。
- **持久层**：`MyBatis-Plus`。相比原生 MyBatis，MP 写单表 CRUD 极快，能省很多手写 SQL 的时间。
- **数据库**：`MySQL`。
- **中间件（为第二阶段秒杀预留）**：
  - **Redis**：后续用来做库存预热缓存、分布式锁以及接口限流。
  - **消息队列**：打算上 RabbitMQ 或 RocketMQ，用来做异步下单，帮数据库削峰。
- **接口文档**：集成了 `Springdoc OpenAPI / Swagger`，方便本地调试和前后端对接口。
